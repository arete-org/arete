## What changed
- Reworked [docs/README.md](./docs/README.md) into a newcomer-first entrypoint with a clear "start here" path and direct links to architecture, status, API, AI guide, decisions, and proposals.
- Added [docs/architecture/README.md](./docs/architecture/README.md) with a practical reading order:
  - core architecture path for first-time readers
  - adjacent docs for important context
  - supplemental docs for deeper, non-blocking detail
- Added [docs/status/README.md](./docs/status/README.md) to separate:
  - active status documents (read first)
  - `status/archive` historical rollout context (read second)
- Normalized status filenames from `YYYY-MM-DD-*` to `YYYY-MM-*` in both:
  - `docs/status/`
  - `docs/status/archive/`
- Updated affected internal links in architecture/decisions/status docs so navigation remains correct after the renames.

## Why
The task focus was navigation clarity, not adding more process metadata. New contributors could find docs, but it was not obvious what to read first versus what is historical context. This PR addresses that by strengthening index pages and link flow, while avoiding per-file governance/freshness banners and avoiding broad doc rewrites.

## Important implementation details
- Classification and orientation are kept in index pages (`docs/README.md`, `docs/architecture/README.md`, `docs/status/README.md`) rather than per-file labels.
- The status timestamp normalization was intentionally low-churn:
  - only the listed status/archive docs were renamed
  - no API/openapi files were renamed
  - body text was only touched where needed to fix references
- Existing historical context is preserved in `docs/status/archive`; it is now positioned more clearly as historical rollout context rather than first-stop architecture guidance.
